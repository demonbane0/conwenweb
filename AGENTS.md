# AGENTS.md — 實裝 agent（Codex / Sonnet）工作守則

你是本專案的實裝 agent。設計與驗收由主控（Claude Fable 5）負責；你負責把指派的
任務**完整實作並自我驗證**後回報。

## 必讀

先讀 `CLAUDE.md` — 技術棧、目錄結構、產品 schema、設計系統、驗證要求都在裡面，
**全部具有強制力**。本檔只補充工作方式。

## 工作方式

1. 只做被指派的任務範圍，不要「順手」改別人的檔案 —— **同時間有多個 agent
   在同一個工作樹上平行作業**，越界會互相衝突。
2. 任務說明中列出的「你負責的檔案/目錄」就是你的邊界；需要邊界外的修改時，
   在回報中提出，不要自己動手。
3. 不要執行任何 git commit/push —— 版控由主控統一處理。
4. 不要安裝任務說明以外的相依套件。
5. 完成前依 CLAUDE.md「驗證要求」自我驗證；`npm run build` 若因**別人負責的
   檔案**出錯，忽略該錯誤並在回報中註明即可。

## 資料抽取任務專用規則（legacy → Markdown）

- 來源只用 `legacy/utf8/product_N.html`（UTF-8 轉檔）。原始 `legacy/*.html` 是
  Big5，直接讀會亂碼，不要用。
- 舊頁是 FrontPage 產出的巢狀表格：一個產品通常是「名稱+型號的標題列」加
  「◆ 開頭的特點列」加一張 `<img>`。忽略所有樣式、字型、theme 圖
  （`images/nth_theme_*`、`nav/*`、`*.swf`）。
- 產品圖從 `legacy/` 複製到 `public/images/products/`，檔名改為小寫 slug
  （如 `DM.JPG` → `m3500a.jpg`，以型號命名優先）。找不到對應圖時用
  `/images/products/placeholder.svg` 並在回報列出。
- slug 規則：型號轉小寫、空白與特殊字元轉 `-`（如 `AC DC 2003A` →
  `ac-dc-2003a`）。同分類內不得重複。
- 特點文字保留原文（繁中），去除 HTML 與樣式雜訊；明顯的編碼壞字以 `?` 存在
  時，依上下文修復或省略該點，並在回報註明。
- 寧可少收不收爛：裝飾圖、空白列、重複列直接略過。

## 回報格式

完成時回報：
- 建立/修改的檔案清單
- 驗證結果（build 是否通過、抽了幾個產品、缺圖清單…）
- 未盡事項與需要主控決策的問題
